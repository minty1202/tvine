# #26 ステータス管理 v4（PTY 出力監視 + 3 状態遷移）

## 経緯

v1（hooks）→ v2（PTY 2 状態）→ v3（hooks Waiting 廃止）と試行錯誤した結果、以下の結論に至った：

- hook ベースは AskUserQuestion 後のギャップがあり、Claude Code の hook イベントへの依存が高い
- OSC タイトルパースは Claude Code の内部実装への依存が高い（ccmanager 方式）
- PTY 出力監視が最も安定。キー入力エコーの誤検知は 3 状態遷移（Evaluating 状態）で吸収する

v2 のコード（useSessionTerminal の DI リファクタ、StatusIndicator、statusStore）をベースに、ptyActivityMonitor を 3 状態対応に拡張する。バックエンド（Plugin 生成、Unix socket 等）は一切不要。

## 完了時の状態

- セッション一覧の各カードにステータス（Evaluating / Busy / Idle）が表示される
- PTY 出力開始で Evaluating、出力が続けば Busy、出力が途切れれば Evaluating に戻り、沈黙 2 秒で Idle
- Claude Code がクラッシュした場合は PTY exit を検出して Idle に戻る
- tvine 再起動時は全セッション Idle

## 方針

### ptyActivityMonitor を 3 状態対応に拡張

#### 方針

v2 の ptyActivityMonitor（即 Busy + 沈黙で Idle）を、Evaluating を経由する 3 状態遷移に変更する。

#### 実装方法

状態遷移のロジック:

```
onPtyOutput(sessionId):
  reset silenceTimer(2s → Idle)

  if Idle:
    set Evaluating
    record spinnerStartTime

  if Evaluating:
    if now - spinnerStartTime > BUSY_THRESHOLD:
      set Busy
    reset gapTimer

  if Busy:
    reset gapTimer(GAP_THRESHOLD → Evaluating)

gapTimer fires:
  Busy → Evaluating

silenceTimer fires:
  → Idle
```

- `BUSY_THRESHOLD`: Evaluating → Busy の昇格閾値（要実測で決定）
- `GAP_THRESHOLD`: Busy → Evaluating の降格閾値（連続出力の途切れ検出）
- `SILENCE_THRESHOLD`: 2 秒（Evaluating → Idle）

callbacks を `onBusy` / `onIdle` から `onEvaluating` / `onBusy` / `onIdle` に拡張する。

### StatusIndicator を 3 状態対応に更新

#### 方針

SessionStatus 型に Evaluating を追加し、StatusIndicator で 3 状態を表示する。

#### 実装方法

- `src/stores/statusStore.ts` の `SessionStatus` に `'Evaluating'` を追加
- StatusIndicator の `statusColor` と `statusChar` に Evaluating を追加
- Evaluating: ブレイルスピナーアニメーション（現在の Busy と同じ見た目を流用）
- Busy: 新規デザイン（別途決定）

### Busy の表示デザインを決定

#### 方針

Evaluating（くるくる）と視覚的に区別できる Busy のデザインを決定する。

#### 候補

実装時に決定する。

### バックエンドの hook インフラを削除

#### 方針

v1 で追加したバックエンドのステータス検出インフラを削除する。PTY 出力監視はフロントエンドで完結するため不要。

#### 実装方法

削除対象:
- `crates/data/src/manage/plugin.rs` — Plugin ファイル生成
- `crates/kernel/src/model/status.rs` — SessionStatus enum（TS 側で手書き定義に移行済み）
- `src-tauri/src/routes/status.rs` — Unix socket リスナー
- `src-tauri/src/lib.rs` — Plugin 生成呼び出し、socket リスナー起動
- `crates/pty/src/process.rs` — `--plugin-dir` と環境変数の注入
- 関連する trait シグネチャの引数（`socket_path` 等）

`src/generated/SessionStatus.ts` も不要（`src/stores/statusStore.ts` で手書き定義）。
