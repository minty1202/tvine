## 方針

### 初期化のタイミング

#### 方針

- アプリ起動時に自動で初期化する。明示的な init コマンドは設けない
- `~/.tvine/` と `~/.tvine/projects/<project-id>/` の両方を起動時に「なければ作る」
- カレントディレクトリがプロジェクト内（git リポジトリ内）であることを前提とする。パス引数によるプロジェクト指定は行わない（複雑になるため、必要になったら追加する）
- 既にディレクトリが存在する場合は何もしない（冪等）
- プロジェクトディレクトリを `mv` で移動した場合、旧パスのデータは孤立する（旧 project-id のデータが残る）。移動先では新しいプロジェクトとして初期化し直す。旧データは手動で削除する想定

#### 起動フロー

起動フローは `api::operator` が組み立てる（handler がアプリ動作中のリクエストに応えるのに対し、operator はアプリ自体に対する操作を担う）。

1. バリデーション（`crates/exec/` — 外部バイナリの存在確認を担うクレート）
   - `git` と `claude` コマンドが使えるか
2. `~/.tvine/` ルートディレクトリの作成（アプリレベル、プロジェクトに依存しない）
3. プロジェクトルートの取得
   - git リポジトリ内であること（ビジネスルールとして kernel で定義）
   - `crates/client/git/` が git2 を使ってリポジトリを探索し、メインのリポジトリパスを返す
   - git 操作は git 操作クレート（git2 のラッパー + git2 にない操作は `std::process::Command`）が提供し、adapter がそれを使って kernel の trait を実装する
4. プロジェクトディレクトリの作成（プロジェクトレベル）
   - プロジェクト識別子の生成、ディレクトリ構造の作成、project.json の生成
5. メインの処理を実行

#### CLI での確認

- 初期化は全コマンド実行前に自動で行われるため、`init` サブコマンドは不要
- 実験用に `teardown` サブコマンドを用意し、`~/.tvine/` を丸ごと削除できるようにする

### client/git のエラー型

#### 方針

- client/git 専用のエラー型を定義し、git2::Error を外部に漏らさない
- adapter のテスト時に git client をモックする際、エラーケースを簡単に構築できるようにする
- 最初は `NotARepository` 程度のシンプルな variant から始め、git 操作が増えた時に追加する

### プロジェクト識別子

#### 方針

- git ルートパスの `/` を `-` に置き換えたものを識別子とする（例: `/Users/aki/dev/myapp` → `-Users-aki-dev-myapp`）
- Claude Code と同じ方式。ハイフンを含むディレクトリ名との衝突は理論上あり得るが、現実的に問題になるケースは稀なので許容する

### 初期化で作成するもの

#### 方針

- 初期化ではディレクトリ構造と `project.json` を作成する。セッション等のファイルは必要になった時に作る
- plugin ディレクトリはこの Issue のスコープ外

#### 作成するもの

- `~/.tvine/` — ルートディレクトリ
- `~/.tvine/projects/<project-id>/` — プロジェクトディレクトリ
- `~/.tvine/projects/<project-id>/sessions/` — セッション格納場所
- `~/.tvine/projects/<project-id>/project.json` — プロジェクトのメタ情報。初期化時に `pwd`（プロジェクトルートパス）を保存する
